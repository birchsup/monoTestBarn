# PRODUCT_BACKLOG

Документ составлен на основе `docs/API_CONTRACT.md` и текущего состояния API (handlers/routes/DB).

## 🔹 MVP задачи (обязательно для рабочего продукта)

### 1. Единый API-контракт для test cases и test suites
- Название: Канонизация REST-маршрутов v1
- Тип: API / Backend
- Описание проблемы: В API смешаны разные стили (`/testcase?id=...`, `/test-case/update?id=...`, `/testcases/{id}`), что ломает предсказуемость и усложняет клиентов.
- Что нужно реализовать: Принять единый resource-oriented контракт (`/testcases/{id}`, `/test-suites/{id}` и т.д.), зафиксировать его в документации и синхронизировать роуты.
- Почему это важно для MVP: Без стабильного контракта невозможно надёжно строить UI и автотесты.
- Критерии готовности (DoD):
  - Все CRUD-операции test cases/test suites используют единый стиль URL.
  - Контракт задокументирован и совпадает с реализацией.
  - Интеграционные тесты проходят на новом контракте.

### 2. Полноценный MVP для test runs
- Название: Включить API создания и чтения тест-ранов
- Тип: Backend / API / DB
- Описание проблемы: Таблицы `test_runs` и `test_run_cases` есть, но публичный API для прогонов фактически отключён.
- Что нужно реализовать: Активировать и завершить endpoint-ы минимум для `POST /test-runs`, `GET /test-runs`, `GET /test-runs/{id}` с привязкой кейсов.
- Почему это важно для MVP: Продукт не выполняет ключевой сценарий «запустить набор тестов».
- Критерии готовности (DoD):
  - Можно создать прогон по test suite.
  - Можно получить список прогонов и детали прогона.
  - Данные прогона сохраняются в БД и возвращаются API.

### 3. Отметка прохождения тестов в рамках прогона
- Название: Статусы выполнения test case в run
- Тип: Backend / API / DB
- Описание проблемы: Нет механизма фиксировать результат каждого кейса в конкретном прогоне (Passed/Failed/Blocked/Skipped).
- Что нужно реализовать: Расширить модель `test_run_cases` (статус, комментарий, время выполнения, исполнитель) и добавить endpoint-ы обновления статуса кейса в run.
- Почему это важно для MVP: Без отметки результатов система не решает задачу управления ручным тестированием.
- Критерии готовности (DoD):
  - Для каждого кейса в run можно сохранить и изменить статус.
  - Доступен итог прогона (сводка по статусам).
  - API возвращает актуальные статусы в деталях run.

### 4. Единый формат ошибок и валидации
- Название: Стандартизация ошибок API
- Тип: API / Backend / UX
- Описание проблемы: Ошибки приходят plain text, not found обрабатывается непоследовательно (часть `500` вместо `404`).
- Что нужно реализовать: Ввести JSON-формат ошибок (`code`, `message`, `details`), единые правила статусов `400/404/409/500`, валидацию входных данных.
- Почему это важно для MVP: UI и автотесты не могут стабильно обрабатывать ошибки без стандартного формата.
- Критерии готовности (DoD):
  - Все handlers возвращают JSON-ошибки.
  - `sql.ErrNoRows` везде мапится на `404`.
  - Документированы и покрыты тестами негативные сценарии.

### 5. Корректная семантика обновления/удаления
- Название: Обработка несуществующих ID через RowsAffected
- Тип: Backend / API
- Описание проблемы: `PUT/DELETE` могут возвращать успех даже если сущность отсутствует.
- Что нужно реализовать: Проверять `RowsAffected` и возвращать `404` для несуществующих ресурсов.
- Почему это важно для MVP: Иначе пользователь получает ложный успех и теряет доверие к системе.
- Критерии готовности (DoD):
  - Все update/delete endpoint-ы проверяют фактическое изменение данных.
  - Для несуществующего ID возвращается `404`.
  - Поведение отражено в контракте и тестах.

### 6. Целостность данных и безопасные удаления
- Название: FK-стратегия и каскадность
- Тип: DB / Backend
- Описание проблемы: Удаление suite при связанных кейсах может завершаться ошибкой FK; поведение не зафиксировано как продуктовое правило.
- Что нужно реализовать: Явно определить стратегию (restrict или cascade), привести миграции и API-поведение в соответствие.
- Почему это важно для MVP: Базовые операции управления наборами должны работать предсказуемо.
- Критерии готовности (DoD):
  - Для delete-сценариев определено и реализовано единое правило.
  - Нет «плавающих» ошибок FK без понятного ответа клиенту.
  - Поведение покрыто интеграционными тестами.

### 7. Интеграционные тесты ключевого пользовательского потока
- Название: E2E-цепочка «создал кейс → добавил в suite → создал run → отметил статусы»
- Тип: Tests / Backend
- Описание проблемы: Текущие интеграционные тесты частично рассинхронизированы с API и не покрывают test runs.
- Что нужно реализовать: Переписать/дополнить интеграционные тесты под канонический контракт и полный MVP-флоу.
- Почему это важно для MVP: Без автоматической проверки критичного сценария высокий риск регрессий.
- Критерии готовности (DoD):
  - Тестовый сценарий MVP проходит стабильно.
  - Тесты независимы от абсолютных локальных путей.
  - CI может запускать тесты в стандартном окружении.

### 8. Базовый интерфейс оператора тестирования
- Название: Минимальный UI для ручного тестирования
- Тип: Frontend / UX / API
- Описание проблемы: Даже при рабочем API продукту нужен интерфейс для реального использования тестировщиками.
- Что нужно реализовать: Экран списка test cases, экран suite, создание run, отметка статусов кейсов в run, отображение итогов.
- Почему это важно для MVP: Ценность продукта проявляется в прикладном использовании, а не только в наличии endpoint-ов.
- Критерии готовности (DoD):
  - Пользователь может пройти полный сценарий ручного прогона без прямых вызовов API.
  - Ошибки и статусы отображаются понятно.
  - UX-поток подтверждён минимум 1 демонстрационным сценарием.

## 🔹 UX / Улучшения (после MVP)

### 1. Пагинация списков
- Название: Пагинация для test cases, suites, runs
- Тип: API / Backend / Frontend / UX
- Описание проблемы: С ростом данных списки становятся тяжёлыми и неудобными.
- Что нужно реализовать: Параметры `limit/offset` или cursor-based пагинацию + метаинформацию (`total`, `next`).
- Почему это важно для MVP: Не критично для запуска, но существенно улучшает производительность и управляемость.
- Критерии готовности (DoD):
  - Все крупные списки поддерживают пагинацию.
  - UI умеет перелистывать страницы.
  - Нет деградации времени ответа на больших объёмах.

### 2. Поиск и фильтрация
- Название: Поиск тестов и прогонов по атрибутам
- Тип: API / Backend / Frontend / UX
- Описание проблемы: Пользователь не может быстро найти нужный кейс/набор/прогон.
- Что нужно реализовать: Фильтры по названию, статусу, suite, дате, исполнителю; полнотекстовый поиск по имени/описанию.
- Почему это важно для MVP: Сильно снижает время операционной работы QA-команды.
- Критерии готовности (DoD):
  - Доступны серверные фильтры и поиск.
  - UI предоставляет форму фильтрации.
  - Сценарии поиска покрыты тестами API.

### 3. Сортировка
- Название: Сортировка по ключевым полям
- Тип: API / Frontend / UX
- Описание проблемы: Данные выводятся в одном порядке, не всегда полезном для анализа.
- Что нужно реализовать: Сортировка по `created_at`, имени, последнему запуску, результату.
- Почему это важно для MVP: Улучшает анализ и ежедневную работу без изменения бизнес-логики.
- Критерии готовности (DoD):
  - API принимает `sort_by` и `order`.
  - UI позволяет менять порядок без перезагрузки контекста.
  - Сортировка корректна и стабильна.

### 4. Группировка по наборам и релизам
- Название: Визуальная группировка тестов
- Тип: Frontend / UX / API
- Описание проблемы: При большом количестве кейсов сложно понимать структуру покрытия.
- Что нужно реализовать: Группировка по test suite, релизу, компоненту (при наличии поля).
- Почему это важно для MVP: Ускоряет подготовку прогонов и контроль покрытия.
- Критерии готовности (DoD):
  - В UI доступны режимы группировки.
  - Переход между группами не ломает фильтры/поиск.
  - Группы согласованы с моделью данных.

### 5. Bulk-операции
- Название: Массовые действия над test cases
- Тип: Frontend / API / Backend / UX
- Описание проблемы: Сейчас операции выполняются по одному объекту, что медленно для регресса.
- Что нужно реализовать: Массовое добавление в suite, массовая смена статуса в run, массовое архивирование/удаление.
- Почему это важно для MVP: Экономит время на типовых ручных операциях.
- Критерии готовности (DoD):
  - API поддерживает batch-запросы.
  - UI поддерживает multi-select и подтверждение массовых действий.
  - Логируются частичные ошибки batch-операций.

### 6. Удобные статусы и цветовая индикация
- Название: Стандартизация статусов и визуальных маркеров
- Тип: UX / Frontend
- Описание проблемы: Без унифицированной визуализации сложно быстро оценить состояние прогона.
- Что нужно реализовать: Единая палитра статусов (`Passed`, `Failed`, `Blocked`, `Skipped`, `Not Run`) и сводные индикаторы.
- Почему это важно для MVP: Повышает читаемость отчётов и скорость принятия решений.
- Критерии готовности (DoD):
  - Для каждого статуса задан текст, цвет и иконка.
  - Сводка прогона читается за 1 экран.
  - Цвета доступны и различимы (минимальные требования accessibility).

### 7. Быстрые действия в карточке теста
- Название: One-click actions для оператора
- Тип: Frontend / UX
- Описание проблемы: Частые операции требуют лишних переходов между страницами.
- Что нужно реализовать: Быстрые кнопки: «добавить в suite», «клонировать», «запустить в run».
- Почему это важно для MVP: Снижает число кликов и утомляемость команды.
- Критерии готовности (DoD):
  - Минимум 3 быстрых действия доступны из карточки/строки списка.
  - Действия подтверждаются обратной связью UI.
  - Нет потери данных при отмене.

## 🔹 Недостающие, но очевидные функции

### 1. Версии тест-кейсов
- Тип: Backend / API / DB / UX
- Описание проблемы: Изменения кейсов перезаписываются без контроля версии.
- Что нужно реализовать: Версионирование кейса с возможностью смотреть и использовать конкретную версию в run.
- Почему это важно: Для аудита и воспроизводимости результатов прогона.
- Критерии готовности (DoD):
  - У кейса есть номер версии.
  - Доступна история версий и выбор версии в run.

### 2. История изменений (audit log)
- Тип: Backend / API / DB / UX
- Описание проблемы: Непрозрачно, кто и когда менял тесты/наборы/результаты.
- Что нужно реализовать: Журнал изменений по ключевым сущностям с автором, временем и diff.
- Почему это важно: Требование зрелых QA-процессов и разборов инцидентов.
- Критерии готовности (DoD):
  - Логируются create/update/delete и смена статусов run.
  - История доступна в UI/API.

### 3. Клоны тестов
- Тип: Backend / API / Frontend
- Описание проблемы: Нет быстрого способа создать похожий тест-кейс.
- Что нужно реализовать: Операция clone с редактированием имени/метаданных после копии.
- Почему это важно: Ускоряет подготовку регресса и вариаций сценариев.
- Критерии готовности (DoD):
  - Клон создаётся одной операцией.
  - Сохраняется связь с исходником (опционально).

### 4. Теги и категоризация
- Тип: DB / Backend / API / UX
- Описание проблемы: Недостаточная классификация кейсов по областям/рискам.
- Что нужно реализовать: Система тегов и фильтрация по тегам.
- Почему это важно: Масштабирование тестовой библиотеки.
- Критерии готовности (DoD):
  - Теги можно назначать/снимать массово.
  - Фильтрация по тегам работает в списках.

### 5. Комментарии к прогону
- Тип: Backend / API / Frontend / UX
- Описание проблемы: Негде фиксировать контекст падений/блокеров прямо в run.
- Что нужно реализовать: Комментарии уровня run и уровня case-result.
- Почему это важно: Упрощает коммуникацию QA ↔ Dev и ретроспективу дефектов.
- Критерии готовности (DoD):
  - Комментарии сохраняются с автором и временем.
  - Комментарии видны в деталях прогона.

### 6. Экспорт результатов
- Тип: Backend / API / UX
- Описание проблемы: Нет удобного способа делиться результатами вне системы.
- Что нужно реализовать: Экспорт run-результатов в CSV/JSON (опционально PDF).
- Почему это важно: Нужен для отчётности и интеграции с внешними процессами.
- Критерии готовности (DoD):
  - Экспорт включает summary и детализацию по кейсам.
  - Файл формируется по запросу и доступен для скачивания.
