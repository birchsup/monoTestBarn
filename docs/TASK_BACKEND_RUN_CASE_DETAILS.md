# TASK (Backend): `GET /test-runs/{runId}/cases/{caseId}` — детали кейса в контексте прогона

Статус: **done** (см. также фронтовую часть — `docs/TASK_FRONTEND_RUN_CASE_DETAILS.md`).

- **Название:** `GET /test-runs/{runId}/cases/{caseId}` — детали кейса в контексте прогона
- **Тип:** Backend / API
- **Описание проблемы:** Со страницы прогона нельзя открыть детальный просмотр кейса.
  Единственный источник данных — `GET /test-runs/{id}`, который возвращает весь прогон целиком;
  для deep-link на конкретный кейс внутри прогона фронту приходится грузить и парсить весь run.
  Отдельного эндпоинта «один кейс в прогоне» нет (`backend/internal/api/testRun.go` содержит
  только `PATCH` по этому пути).
- **Что нужно реализовать:**
  - Новый handler `GET /test-runs/{runId}/cases/{caseId}`, возвращающий объект кейса в контексте
    прогона: `case_id`, `test` (полное содержимое кейса — название, шаги, ожидаемый результат),
    `status`, `comment`, `executed_at`, `executed_by`, плюс `run_id`, `suite_id` и `run_details`
    прогона для хлебных крошек.
  - Функция в `backend/db/testRun.go` (по образцу `GetTestRunByID`, но с фильтром по `trc.case_id`);
    ошибки маппить на существующие `ErrTestRunNotFound` / `ErrTestRunCaseNotFound` → `404`
    с кодами `test_run_not_found` / `test_run_case_not_found`.
  - Зафиксировать эндпоинт в `docs/API_CONTRACT.md`.
- **Критерии готовности (DoD):**
  - Эндпоинт возвращает `200` с полными данными кейса и его результатом в прогоне.
  - Несуществующий run или кейс, не входящий в run, → `404` в стандартном JSON-формате ошибок.
  - Покрыто тестами (`go test ./...`), контракт в `docs/API_CONTRACT.md` совпадает с реализацией.

## Замечание

Поле `test` отдаётся JOIN-ом из `test_cases`, то есть это **текущая** версия кейса, а не снапшот
на момент прогона (так же ведёт себя `GET /test-runs/{id}`). Снапшоты кейсов в run — отдельная
возможная задача, в рамках этой не решается.
