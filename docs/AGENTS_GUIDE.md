# AGENTS_GUIDE

## 1. Общее описание проекта
- Сервис `testBarn` — HTTP API для создания и хранения ручных тест-кейсов и их группировки в тест-сьюты.
- Дополнительно в схеме БД предусмотрены сущности тест-ранов (`test_runs`, `test_run_cases`), но API для запусков тестов сейчас неактивен (код закомментирован).
- Точка входа: `main.go`.
- Основные модули:
  - `main.go` — инициализация конфигурации, БД, роутера, CORS и запуск HTTP-сервера на `:8080`.
  - `config/config.go` — чтение `config.yml` через `viper`, установка `DATABASE_URL`.
  - `internal/api/*.go` — HTTP-обработчики (handlers) для test cases и test suites.
  - `db/*.go` — доступ к PostgreSQL (инициализация пула и SQL-операции).
  - `db/migrations/*.sql` — SQL-миграции структуры БД.

## 2. Как запускать проект

### Переменные окружения
- Основная переменная: `DATABASE_URL`.
- Если `DATABASE_URL` не задана явно, `config.InitConfig()` берёт её из `config.yml`:
  - `development.database_url`
  - `test.database_url`

### Зависимости
- PostgreSQL: обязателен для работы API.
- Redis: не используется.
- Docker для приложения: не настроен (в репозитории нет `Dockerfile`/`docker-compose.yml`).

### Локальный запуск
1. Поднять PostgreSQL локально.
2. Создать БД (по умолчанию в `config.yml` для dev: `testbarn`).
3. Применить миграции из `db/migrations` любым удобным инструментом (например, `golang-migrate`).
4. Запустить сервис:
   ```bash
   go run main.go
   ```
5. Сервер слушает `http://localhost:8080`.

## 3. Как запускать тесты
- В проекте есть интеграционные тесты: `tests_test/integrations/main_test.go`.
- Unit-тесты в текущем состоянии не обнаружены.

### Команды
- Через Task:
  ```bash
  task test
  ```
- Напрямую Go:
  ```bash
  go test -v tests_test/integrations/main_test.go
  ```

### Требования для интеграционных тестов
- Нужен Docker Engine (используется `testcontainers-go`, поднимается `postgres:15`).
- Тесты сами выставляют `DATABASE_URL` и выполняют миграции.

## 4. Архитектурные заметки
- Текущая архитектура близка к 2 слоям:
  - Транспортный слой: `internal/api` (HTTP handlers).
  - Инфраструктурный слой: `db` (SQL и работа с пулом Postgres).
- Явного сервисного слоя (`services`) нет: handlers напрямую вызывают функции из `db`.
- Явного разделения на repository-интерфейсы нет: SQL находится в функциях пакета `db`.
- Конфигурация отделена в `config`, но ограничена настройкой БД.

## 5. Ограничения для других AI-агентов
- Тестер:
  - Изменяет только файлы `*_test.go`.
  - Не меняет production-код, миграции и конфиги инфраструктуры.
- Разработчик:
  - Меняет только production-код и связанные артефакты (если это необходимо для фичи/фикса).
  - Не меняет тесты без явной необходимости.
- DevOps-агент:
  - Работает только с infra-файлами (Docker, CI/CD, окружение, деплой).
  - Не меняет бизнес-логику API и SQL-логику приложения.

## 6. Неясности и вопросы
- В интеграционном тесте используется абсолютный путь миграций:
  - `file:///Users/dmitrijsadovnikov/testBarn/db/migrations`
  - Это делает тесты непереносимыми между машинами/путями.
- В `tests_test/integrations/main_test.go` зарегистрирован route `GET /testcases/{id:[0-9]+}` с handler `api.GetTestCase`, но в API-файлах присутствует `GetTestCaseHandler` с query-параметром (`/testcase?id=...`). Требуется сверить актуальный контракт API и тестов.
- В `main.go` для CORS дважды вызывается `handlers.AllowedHeaders(...)`; возможно, вторая настройка перезаписывает или дублирует первую.
- API для `test_runs` в коде присутствует частично, но обработчик создания закомментирован. Нужно уточнить статус функциональности тест-ранов.
- В репозитории нет README с официальным сценарием запуска/миграций; процесс запуска в этой инструкции выведен из текущего кода.
