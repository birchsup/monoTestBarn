## 🔹 Backend задачи

### 1) Канонизация REST-маршрутов v1
- **Название:** Унификация CRUD-роутов для `testcases` и `test-suites`
- **На основе какой продуктовой задачи:** MVP #1 «Канонизация REST-маршрутов v1»
- **Что конкретно реализовать в коде:**
  - Перевести роуты на resource-oriented стиль:
    - `POST /testcases`, `GET /testcases`, `GET /testcases/{id}`, `PUT /testcases/{id}`, `DELETE /testcases/{id}`
    - `POST /test-suites`, `GET /test-suites`, `GET /test-suites/{id}`, `PUT /test-suites/{id}`, `DELETE /test-suites/{id}`
  - Удалить/пометить deprecated legacy-роуты (`/testcase?id=...`, `/test-case/update?id=...` и т.д.).
  - Синхронизировать handlers и API-документацию.
- **Какие API/таблицы затрагиваются:**
  - API: все CRUD endpoint’ы `testcases` и `test-suites`
  - Таблицы: `test_cases`, `test_suites`, `test_suite_cases`
- **Definition of Done:**
  - Все CRUD работают через единый URL-стиль.
  - Документация и код совпадают.
  - Интеграционные тесты на новый контракт зелёные.

### 2) API создания и чтения test runs
- **Название:** MVP API для прогонов
  - **На основе какой продуктовой задачи:** MVP #2 «Включить API создания и чтения тест-ранов»Run создаётся как отдельная сущность в таблице test_runs.
 -   Run может формироваться из:
 -   всего тестового набора (test suite)
 -   списка отдельных тест-кейсов
 -   комбинации набора + дополнительных кейсов
- При создании run:
- все тест-кейсы материализуются в таблице test_run_cases
   - для каждого кейса создаётся отдельная запись со статусом по умолчанию not_run
    - если кейс входит и в suite, и в списке test_case_ids — он добавляется только один раз (дедупликация по test_case_id)

- Правила:
  Должен быть передан хотя бы один источник тестов (suite или список кейсов)
  Если оба переданы — run включает:
  все кейсы из suite
  плюс явно указанные test_case_ids
  Если указанный suite не существует → 404
  Если хотя бы один test_case_id не существует → 404
  Повторяющиеся ID должны быть удалены перед вставкой в test_run_cases
- **Какие API/таблицы затрагиваются:**
  - API: `POST /test-runs`, `GET /test-runs`, `GET /test-runs/{id}`
  - Таблицы: `test_runs`, `test_run_cases`, `test_suites`, `test_suite_cases`, `test_cases`
- **Definition of Done:**
  
  - Run-лист и run-details доступны.
  - Данные корректно сохраняются в БД.

### 3) Статусы выполнения кейсов в run
- **Название:** Управление статусами run-case
- **На основе какой продуктовой задачи:** MVP #3 «Статусы выполнения test case в run»
- **Что конкретно реализовать в коде:**
  - Расширить модель результата кейса в run: `status`, `comment`, `executed_at`.
  - Реализовать `PATCH /test-runs/{runId}/cases/{caseId}` для смены статуса и комментария.
  - Реализовать подсчёт summary по run (`passed/failed/blocked/skipped/not_run`).
- **Какие API/таблицы затрагиваются:**
  - API: `PATCH /test-runs/{runId}/cases/{caseId}`, `GET /test-runs/{id}` (с summary)
  - Таблицы: `test_run_cases`, `test_runs`
- **Definition of Done:**
  - Для кейса в run можно менять статус и комментарий.
  - В деталях run возвращается актуальная summary.
  - Статусы валидируются против разрешённого набора.

### 4) Единый формат ошибок и валидаций
- **Название:** Error contract v1
- **На основе какой продуктовой задачи:** MVP #4 «Стандартизация ошибок API»
- **Что конкретно реализовать в коде:**
  - Ввести единый JSON-ответ ошибки: `{ code, message, details }`.
  - Вынести маппинг ошибок в общий helper/middleware (`400/404/409/500`).
  - Централизовать валидацию query/path/body.
  - Гарантировать маппинг `sql.ErrNoRows -> 404`.
- **Какие API/таблицы затрагиваются:**
  - API: все endpoint’ы
  - Таблицы: не напрямую (изменение обработки ошибок)
- **Definition of Done:**
  - Нет plain-text ошибок в публичном API.
  - Статусы ошибок унифицированы и предсказуемы.
  - Негативные сценарии документированы и протестированы.

### 5) Семантика update/delete через RowsAffected
- **Название:** Корректные 404 для несуществующих ресурсов
- **На основе какой продуктовой задачи:** MVP #5 «Обработка несуществующих ID через RowsAffected»
- **Что конкретно реализовать в коде:**
  - Во всех update/delete операциях проверять `RowsAffected`.
  - Возвращать `404`, если строка не изменена/не удалена.
- **Какие API/таблицы затрагиваются:**
  - API: `PUT/DELETE` для `testcases`, `test-suites`, run-case updates
  - Таблицы: `test_cases`, `test_suites`, `test_suite_cases`, `test_run_cases`, `test_runs`
- **Definition of Done:**
  - Ложных `200`/`204` для отсутствующих ID нет.
  - Поведение отражено в API-контракте.

### 6) FK-стратегия удаления
- **Название:** Явная политика удаления связанных сущностей
- **На основе какой продуктовой задачи:** MVP #6 «FK-стратегия и каскадность»
- **Что конкретно реализовать в коде:**
  - Зафиксировать правило: `RESTRICT` или `CASCADE` для связей suite-case и run-case.
  - Привести SQL-логику и handler-ошибки к одному правилу.
  - Добавить доменный error-code для конфликтов целостности (`409`).
- **Какие API/таблицы затрагиваются:**
  - API: delete endpoint’ы suite/case/run
  - Таблицы: `test_suites`, `test_suite_cases`, `test_runs`, `test_run_cases`, `test_cases`
- **Definition of Done:**
  - Поведение удаления полностью предсказуемо.
  - FK-ошибки не «плавают» и возвращаются в стандартизованном формате.

### 7) Пагинация, фильтрация, сортировка API

  - Добавить фильтры: `name`, `status`, `suite_id`, `date_from/date_to`, `executed_by`.
  - Добавить сортировку: `sort_by`, `order`.
- **Какие API/таблицы затрагиваются:**
  - API: list endpoint’ы `testcases`, `test-suites`, `test-runs`
  - Таблицы: `test_cases`, `test_suites`, `test_runs`, `test_run_cases`
- **Definition of Done:**
  - Параметры работают для всех целевых списков.
  - Контракт метаданных пагинации стабилен.

### 8) Batch/bulk операции
- **Название:** Массовые endpoint’ы
- **На основе какой продуктовой задачи:** UX #5
- **Что конкретно реализовать в коде:**
  - Добавить batch API:
    - массовое добавление кейсов в suite,
    - массовая смена статуса кейсов в run,
    - массовое архивирование/удаление кейсов (если архивирование вводится).
  - Возвращать результат по каждому элементу (partial success/report).
- **Какие API/таблицы затрагиваются:**
  - API: `/test-suites/{id}/cases:batch`, `/test-runs/{id}/cases:batch-update`, `/testcases:batch-*`

- **Definition of Done:**

  - Частичные ошибки репортятся в ответе без потери успешных изменений.

## 🔹 QA задачи

### 1) Unit-тесты
- Валидация DTO/запросов (path/query/body).
- Маппинг ошибок в `code/message/details`.
- Подсчёт summary run по статусам.
- Логика batch-операций (partial success).

### 2) Integration-тесты API
- CRUD `testcases`/`test-suites` на канонических роутах.
- `POST/GET test-runs`, `PATCH run-case status`, run summary.
- `RowsAffected -> 404` для всех update/delete.
- FK-policy сценарии delete (`restrict`/`cascade` по принятому правилу).
- Пагинация/фильтрация/сортировка на списках.
- Clone/tags/comments/export.

### 3) E2E-тесты (UI)
- Полный MVP-сценарий: создать case -> добавить в suite -> создать run -> проставить статусы -> проверить итог.
- Сценарии UX: фильтры, сортировка, bulk-действия, быстрые действия.
- Сценарии зрелости: версия кейса, комментарии, экспорт.

### 4) Негативные сценарии
- Невалидные payload/query/path-параметры -> `400`.
- Несуществующие ID -> `404`.
- FK/бизнес-конфликт -> `409`.
- Конкурентные обновления статусов в одном run.
- Ошибки экспорта/пустые результаты/недопустимые фильтры.

### 5) Нефункциональные проверки
- Smoke на CI для миграций и запуска интеграционных тестов в чистом окружении.
- Проверки backward-compatibility (если оставляем переходный период legacy-роутов).
- Проверка времени ответа list endpoint’ов на объёме тестовых данных.

## 🔹 Зависимости

1. `MVP #1` (канонический API-контракт) блокирует:
   - frontend интеграцию,
   - integration/e2e тесты,
   - обновление API-документации.
2. `MVP #2` (create/list/get test-runs) блокирует `MVP #3` (статусы run-case).
3. `MVP #3` блокирует MVP UI-экран выполнения run и summary.
4. `MVP #4` (формат ошибок) блокирует стабильные негативные тесты и единый UI error-handling.
5. `MVP #5` и `MVP #6` зависят от внедрения `MVP #4` (единые коды/формат ошибок).
6. `MVP #7` (интеграционный поток) выполняется после завершения MVP #1-#6.
7. `MVP #8` (базовый UI) начинается после готовности минимального backend: #1, #2, #3, #4.
8. UX-улучшения (#1-#7) выполняются после стабильного MVP и базового UI.
9. Недостающие функции (#1-#6) выполняются волнами после UX-ядра; приоритетно: версии кейсов -> audit log -> теги/комментарии -> экспорт/клоны.

2. S2: MVP #2 + DB миграции под run
3. S3: MVP #3 + MVP #5 + MVP #6
4. S4: MVP #8 (базовый UI) + MVP #7 (e2e-цепочка)
5. S5+: UX #1-#7
6. S6+: Недостающие #1-#6
