# Демо: ценность фронтенд-тестов (unit + скриншот)

Шпаргалка для показа. Аудитория — лиды фронта, менеджеры, тестировщики.
Цель: на живых примерах показать, как тесты ловят регрессии **до прода**.
Тайминг: **10–15 минут**.

Два сценария:

- **Демо 1 — Unit-тесты валидации поля** ловят сломанную логику в CI (GitHub Actions) ещё на этапе PR.
- **Демо 2 — Скриншот-тест (Playwright)** ловит CSS-регрессию, которую обычные тесты не видят.

---

## Что где лежит

| Что | Путь |
|-----|------|
| Логика валидации поля «Name» | `frontend/src/createTestCase/validateName.js` |
| Unit-тесты валидации (чистые) | `frontend/src/createTestCase/validateName.test.js` |
| Тесты валидации в UI (компонент) | `frontend/src/createTestCase/CreateTestCase.test.js` |
| GitHub Actions workflow | `.github/workflows/test.yml` |
| Скриншот-тест Playwright | `frontend/e2e/createTestCase.spec.js` |
| Baseline-скриншоты | `frontend/e2e/createTestCase.spec.js-snapshots/` |
| Готовый «ломающий» CSS-патч | `docs/demo/break-css.patch` |
| Ветка со сломанной валидацией | `demo/broken-validation` |

> Все команды фронта запускаются из каталога `frontend/`.

---

## Подготовка перед показом (один раз, заранее)

```bash
cd frontend
npm install                 # зависимости (включая vitest и @playwright/test)
npx playwright install chromium   # браузер для скриншот-теста (если ещё не стоит)
```

Прогрев — убедиться, что всё зелёное:

```bash
npm run test:unit:run       # 13 тестов зелёные
npm run test:e2e            # 2 скриншот-теста зелёные
```

## Демо 1 — Unit-тесты валидации в GitHub Actions (~6–7 мин)

**Идея, которую продаём:** сломанная логика валидации не доезжает до прода —
красный чек в PR блокирует мёрж.

### Что говорить (вступление, 30 сек)
> «Поле "Name" в форме создания тест-кейса имеет правила: обязательное и не длиннее
> 200 символов. Эти правила покрыты unit-тестами. На каждый push и pull request
> GitHub Actions сам прогоняет их. Сейчас сломаю правило — и увижу красный чек
> ещё до ревью, без ручной проверки.»

### Шаги показа

1. **Стартовое состояние — всё зелёное.**
   Открыть в браузере вкладку **Actions** репозитория → последний прогон на `main` зелёный.
   Локально для наглядности можно показать:
   ```bash
   cd frontend && npm run test:unit:run
   ```
   → `Tests 13 passed`.

2. **Сломать логику валидации.** Переключиться на заранее готовую ветку:
   ```bash
   git checkout demo/broken-validation
   ```
   В ней из `validateName.js` убрана проверка обязательности поля (правило «Name is required»).
   Показать diff одной строкой:
   ```bash
   git diff main -- frontend/src/createTestCase/validateName.js
   ```

3. **Открыть Pull Request** `demo/broken-validation → main` (или он уже открыт заранее).
   В PR во вкладке **Checks** появляется жёлтый «крутится» → затем **красный** ❌
   `Vitest (validation)`. Кликнуть в чек → видно, какой именно тест упал и почему.

4. **Откат → зелёный.** Закрыть PR / переключиться обратно:
   ```bash
   git checkout main
   npm run test:unit:run      # снова 13 passed
   ```
   Чек на `main` — зелёный ✅.

### Месседж для аудитории
> «Регрессию поймал робот за секунды, в PR, до мёржа. Никто руками форму не кликал.
> Красный чек — это стоп-кран перед продом.»

## Демо 2 — Скриншот-тест на Playwright: CSS-регрессия (~5–6 мин)

**Идея, которую продаём:** unit-тесты зелёные, а вёрстка поехала. Скриншот-тест
ловит то, что логические тесты не видят.

На форме есть зелёный бейдж **DEMO** рядом с кнопками — это якорь, чтобы быстро
найти место поломки.

### Шаги показа

1. **Baseline зафиксирован, тест зелёный.**
   ```bash
   cd frontend && npm run test:e2e
   ```
   → `2 passed`. Сказать: «Эталонный скриншот формы сохранён в репозитории.»

2. **Сломать CSS** заранее готовым патчем (мгновенно):
   ```bash
   git apply docs/demo/break-css.patch
   ```
   (Патч сдвигает кнопки и перекрашивает бейдж DEMO — заметное смещение вёрстки.)

3. **Запустить тест → падение с visual diff:**
   ```bash
   npm run test:e2e
   ```
   → тест красный. Сказать: «Логика не менялась, тесты валидации всё ещё зелёные —
   а вот вид формы изменился, и скриншот-тест это поймал.»

4. **Открыть HTML-репорт и показать сравнение:**
   ```bash
   npm run test:e2e:report
   ```
   В отчёте у упавшего теста — три картинки: **Expected** (baseline) / **Actual** /
   **Diff** (розовым подсвечена разница). Это самый эффектный момент — показать diff.

5. **Откат → зелёный:**
   ```bash
   git apply -R docs/demo/break-css.patch
   npm run test:e2e            # снова 2 passed
   ```

### Месседж для аудитории
> «Это та регрессия, которую на код-ревью почти не видно и юнит-тесты пропускают.
> Скриншот-тест сравнивает картинку с эталоном и падает на любом визуальном сдвиге.»

## Тайминг (ориентир)

| Блок | Время |
|------|-------|
| Вступление: о боли с валидацией и регрессиями | 1 мин |
| Демо 1: зелёный → красный PR → откат | 6–7 мин |
| Демо 2: baseline → сломали CSS → diff → откат | 5–6 мин |
| Вопросы / вывод | 1–2 мин |

## Шпаргалка команд (всё из `frontend/`)

```bash
# Демо 1 (unit)
npm run test:unit:run
git checkout demo/broken-validation   # сломать
git checkout main                     # вернуть

# Демо 2 (скриншот)
npm run test:e2e                      # прогнать
git apply   docs/demo/break-css.patch # сломать CSS
git apply -R docs/demo/break-css.patch# вернуть
npm run test:e2e:report               # открыть отчёт с diff
```

> После демо можно убрать видимый бейдж `DEMO` из формы (`CreateTestCase.js` +
> `createTestCase.css`) — он добавлен только как визуальный якорь для показа.
